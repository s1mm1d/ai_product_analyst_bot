## Описание проекта

Этот n8n workflow реализует Telegram-бота-аналитика, который принимает описание аналитической задачи в свободной форме и возвращает полноценный структурированный план продуктового исследования.

В основе — **многоэтапный LLM-пайплайн** из 8 последовательных агентов (Mistral AI), каждый из которых отвечает за конкретную часть плана: от формулировки проблемы до критического обзора с оценкой. Все выходы агентов валидируются через **JSON Schema** (Structured Output Parsers), что гарантирует предсказуемый и пригодный для форматирования результат.

Дополнительно workflow включает **Offline Evaluator** — отдельный агент, который сравнивает сгенерированный ответ с эталонными ответами из встроенного датасета и выставляет оценку по 5 критериям.

### Ключевые возможности

- Анализ задач в свободной форме на любом языке (бот автоматически отвечает на языке запроса)
- Многоэтапный пайплайн с 8 специализированными агентами
- Структурированные выходы через JSON Schema на каждом этапе
- Проверка качества запроса перед запуском пайплайна (валидация через LLM)
- Трекинг состояния пользователей между сессиями через Data Tables
- Автоматическое разбиение длинных ответов на части (лимит Telegram — 3800 символов)
- Offline-оценка качества сгенерированного плана с вердиктом
- Полное логирование в Data Table: ответ, оценки обоих эвалюаторов, метаданные

## Архитектура workflow

Workflow состоит из **52 узлов**, сгруппированных в четыре логических блока:

```
[Telegram Trigger]
       │
[Extract Message Data]
       │
[Filter Bot Commands] ──── /start, /help ──── [Send Bot Description]
       │
[Check Message Type]
    ├── /form_problem ──── [Record Command Usage] ──── [Send Bot Description]
    └── (другое сообщение)
           │
     [Check User State]
           │
     [Has Used Command Before?]
          ├── Нет ──── [Send Command Required Message]
          └── Да
                │
         [Frame Problem] ──── [Check Request Clarity]
                                   ├── Нечёткий ──── [Request Clarification]
                                   └── Чёткий
                                          │
                              [Send Processing Notification]
                                          │
                              ┌─────────────────────────┐
                              │  АНАЛИТИЧЕСКИЙ ПАЙПЛАЙН │
                              │  (8 агентов Mistral AI) │
                              └─────────────────────────┘
                                          │
                               [Format Response]
                                          │
                              ┌───────────────────────┐
                              │   [Split Long Message] │
                              │   [Send to Telegram]   │
                              └───────────────────────┘
                                          │
                              ┌───────────────────────┐
                              │  [Dataset JSON]        │
                              │  [Offline Evaluator]   │
                              │  [Merge Results]       │
                              │  [Log to Data Table]   │
                              └───────────────────────┘
```

## Как работает бот

### Команды бота

| Команда | Действие |
|---|---|
| `/start` | Отправляет описание возможностей бота |
| `/help` | Отправляет описание возможностей бота |
| `/form_problem` | Активирует режим ввода задачи + описание бота |
| Любой текст после `/form_problem` | Запускает аналитический пайплайн |

### Логика взаимодействия

#### 1. Первое обращение (без команды)

Если пользователь отправляет произвольное сообщение без предварительного использования `/form_problem`, бот возвращает:

> ⚠️ Пожалуйста, сначала используйте команду `/form_problem` для получения инструкций по работе с ботом.

#### 2. Команда `/form_problem`

Бот:
- Записывает состояние `last_command = /form_problem` в таблицу `user_status`
- Отправляет описание своих возможностей и список примеров запросов
- Ожидает следующего сообщения с задачей

#### 3. Описание аналитической задачи

Бот:
- Проверяет, что пользователь ранее использовал `/form_problem`
- Через агент **Frame Problem** оценивает чёткость задачи
  - Если задача нечёткая → отправляет запрос на уточнение с подсказкой от LLM
  - Если задача чёткая → отправляет уведомление `⏳ Начинаю анализ...` и запускает полный пайплайн
- Возвращает структурированный план анализа

#### 4. Обработка ошибок и нечётких запросов

Если Frame Problem решает, что запрос недостаточно конкретен (например, просто «анализ» или «помоги»), бот отвечает:

> ❓ ТРЕБУЕТСЯ УТОЧНЕНИЕ  
> *[объяснение от LLM, что именно нужно уточнить]*  
> Пожалуйста, укажите: конкретную бизнес-цель, что именно анализировать, какой результат ожидается.

## Аналитический пайплайн (8 этапов)

Все 8 агентов работают последовательно, передавая результат следующему. Каждый агент оснащён отдельным экземпляром **Mistral Model** и **Structured Output Parser** с JSON Schema.

Все агенты автоматически отвечают на языке пользовательского запроса (`RESPOND IN THE SAME LANGUAGE as the input text`).

---

### Этап 1 — Frame Problem (`Mistral Model - Frame`)

**Роль агента:** Валидация запроса и формулировка структурированной постановки проблемы.

**Что делает:**
- Оценивает, достаточно ли информации в запросе для анализа
- Извлекает: `business_goal`, `analysis_goal`, `task_type`, `key_entities`, `assumptions`
- Устанавливает флаг `is_clear` (true/false), который управляет дальнейшим flow
- При нечётком запросе формирует `clarification_needed` — конкретное объяснение, что нужно уточнить

**Модель:** `ministral-8b-latest` (Frame), остальные агенты используют модель по умолчанию из credentials.

**Схема выхода (JSON Schema):**
```json
{
  "is_clear": boolean,
  "business_goal": string,
  "analysis_goal": string,
  "task_type": string,
  "key_entities": [string],
  "assumptions": [string],
  "clarification_needed": string
}
```

---

### Этап 2 — Generate Hypotheses (`Mistral Model - Hypotheses`)

**Роль агента:** Генерация тестируемых гипотез на основе постановки проблемы.

**Что делает:**
- Формирует список гипотез с детальным обоснованием каждой
- Объясняет, почему гипотеза важна и как связана с бизнес-целью

**Входные данные:** результаты Frame Problem (business_goal, task_type, key_entities)

**Схема выхода:**
```json
{
  "hypotheses": [
    {
      "hypothesis": string,
      "rationale": string
    }
  ]
}
```

---

### Этап 3 — Select Metrics (`Mistral Model - Metrics`)

**Роль агента:** Подбор метрик для измерения результата анализа.

**Что делает:**
- Определяет **primary metrics** (ключевые метрики цели)
- Определяет **secondary metrics** (вспомогательные метрики)
- Определяет **guardrail metrics** (метрики-ограничители, которые не должны ухудшиться)
- Даёт чёткое определение каждой метрики

**Схема выхода:**
```json
{
  "primary_metrics": [{"name": string, "definition": string}],
  "secondary_metrics": [{"name": string, "definition": string}],
  "guardrail_metrics": [{"name": string, "definition": string}]
}
```

---

### Этап 4 — Identify Data Requirements (`Mistral Model - Data`)

**Роль агента:** Определение требований к данным для проведения анализа.

**Что делает:**
- Определяет источники данных (data sources)
- Указывает необходимые таблицы и поля
- Описывает события (events), которые нужно трекать
- Задаёт временные окна (time windows) для анализа

---

### Этап 5 — Define Segments (`Mistral Model - Segments`)

**Роль агента:** Предложение релевантных сегментов пользователей для анализа.

**Что делает:**
- Формирует список сегментов на основе задачи, типа задачи и гипотез
- Выделяет приоритетные сегменты, наиболее важные для понимания проблемы

**Входные данные:** текст задачи + task_type + список гипотез из этапа 2

---

### Этап 6 — Create Analysis Plan (`Mistral Model - Plan`)

**Роль агента:** Создание пошагового плана выполнения анализа.

**Что делает:**
- Генерирует ровно **5–7 чётких, actionable шагов**
- Каждый шаг конкретен, с описанием действия и ожидаемого инсайта
- Нумерует шаги

**Входные данные:** исходная задача + business_goal + гипотезы + primary metrics + сегменты

---

### Этап 7 — Suggest Visualizations (`Mistral Model - Viz`)

**Роль агента:** Подбор визуализаций для представления результатов.

**Что делает:**
- Предлагает ровно **5 наиболее важных визуализаций**
- Объясняет цель каждой визуализации и какие инсайты она раскрывает

**Входные данные:** задача + primary metrics + secondary metrics + план анализа

---

### Этап 8 — Review Plan (`Mistral Model - Review`)

**Роль агента:** Критический обзор и оценка всего сгенерированного плана.

**Что делает:**
- Оценивает план по шкале **0–10** по следующим критериям:
  - `0–2`: отсутствуют ключевые разделы
  - `3–4`: есть проблема и гипотезы, но нет метрик
  - `5–6`: все разделы есть, но поверхностно
  - `7–8`: все разделы хорошего качества
  - `9–10`: идеально с временными рамками и сегментами
- Возвращает JSON: `{"score": число, "comment": "текст"}`

**Входные данные:** полный контекст всех предыдущих 7 этапов

## Offline Evaluator

Это отдельный LLM-агент, который запускается **параллельно** с отправкой ответа пользователю (после Format Response) и оценивает качество сгенерированного плана путём сравнения с эталонными ответами из встроенного датасета.

### Как работает

1. **Dataset JSON** — узел с встроенным JSON-датасетом эталонных пар `prompt → ideal_answer`
2. **Offline Evaluator** — агент находит в датасете наиболее похожий prompt и сравнивает `model_answer` с `ideal_answer`
3. **Оценка** выставляется по 5 критериям:

| Критерий | Максимум | Описание |
|---|---|---|
| `structure` | 2 | Структурированность и логичность ответа |
| `completeness` | 3 | Полнота покрытия всех аспектов задачи |
| `hypotheses_quality` | 2 | Качество и релевантность гипотез |
| `actionability` | 2 | Практичность и применимость рекомендаций |
| `relevance` | 1 | Соответствие исходному запросу |

**Максимальный суммарный балл: 10**

4. Возвращает JSON:
```json
{
  "match_quality": 0-10,
  "total_score": 0-10,
  "verdict": "excellent" | "good" | "acceptable" | "weak" | "bad",
  "scores": {
    "structure": 0-2,
    "completeness": 0-3,
    "hypotheses_quality": 0-2,
    "actionability": 0-2,
    "relevance": 0-1
  },
  "comment": "краткий комментарий"
}
```

5. Все оценки обоих эвалюаторов (Review + Offline) логируются в Data Table.

### Расширение датасета

Датасет хранится прямо в узле **Dataset JSON** в формате JSON. Чтобы добавить новые эталонные примеры, откройте узел и добавьте объекты в массив `dataset`:

```json
{
  "dataset": [
    {
      "prompt": "Описание аналитической задачи...",
      "ideal_answer": "📊 ANALYTICS PLAN\n🎯 PROBLEM\nGoal: ..."
    }
  ]
}
```

## Формат ответа

Финальный ответ пользователю начинается с заголовка `📊 ANALYTICS PLAN` и содержит все 7 разделов плана:

```
📊 ANALYTICS PLAN

🎯 PROBLEM
Goal: [бизнес-цель]
Type: [тип задачи]

💡 HYPOTHESES (N)
1. [гипотеза 1]
2. [гипотеза 2]
...

📊 KEY METRICS
Primary: [список]
Secondary: [список]
Guardrail: [список]

🗄️ DATA REQUIREMENTS
[требования к данным]

👥 SEGMENTS
[сегменты]

📋 ANALYSIS PLAN
[пошаговый план]

📈 VISUALIZATIONS
[визуализации]

⭐ REVIEW
Score: X/10
[комментарий]
```

Если ответ превышает **3800 символов**, он автоматически разбивается на части и отправляется несколькими сообщениями с пометкой `Part N/Total:`.

## Установка и настройка

### Шаг 1: Импорт workflow

1. Откройте n8n
2. Перейдите в раздел **Workflows**
3. Нажмите **Import from File**
4. Выберите файл `Telegram_Product_Analytics_Assistant_with_Multi-Stage_LLM_Analysis.json`

Workflow будет импортирован со всеми 52 узлами, связями и настройками.

---

### Шаг 2: Настройка Telegram

1. Создайте бота через **@BotFather** в Telegram
2. Получите **API токен** бота
3. В workflow откройте узел **Telegram Trigger**
4. Создайте новые Telegram credentials или выберите существующие
5. Введите полученный API токен
6. Повторите для всех узлов типа `Send to Telegram` (их 4 штуки):
   - **Send Bot Description**
   - **Send Command Required Message**
   - **Request Clarification**
   - **Send Processing Notification**
   - **Send to Telegram** (основная отправка)

---

### Шаг 3: Настройка Mistral AI

1. Получите **API ключ** на [console.mistral.ai](https://console.mistral.ai)
2. В workflow найдите все узлы с **Mistral Model** (их 9 штук):

| Узел | Используется в этапе |
|---|---|
| Mistral Model - Frame | Frame Problem |
| Mistral Model - Hypotheses | Generate Hypotheses |
| Mistral Model - Metrics | Select Metrics |
| Mistral Model - Data | Identify Data Requirements |
| Mistral Model - Segments | Define Segments |
| Mistral Model - Plan | Create Analysis Plan |
| Mistral Model - Viz | Suggest Visualizations |
| Mistral Model - Review | Review Plan |
| Mistral Model - Offline Evaluator | Offline Evaluator |

3. В каждом узле:
   - Откройте узел
   - Создайте или выберите **Mistral AI credentials**
   - Введите ваш API ключ
   - Нажмите **Save**

> **Примечание:** Узел `Mistral Model - Frame` использует модель `ministral-8b-latest` (лёгкая модель для быстрой валидации). Остальные узлы — модель по умолчанию из credentials (рекомендуется `mistral-large-latest`).

---

### Шаг 4: Создание Data Tables

Важно: таблицы данных должны быть созданы вручную перед активацией workflow. Перейдите в раздел **Data Tables** вашего n8n инстанса.

---

#### Таблица 1: Трекинг состояния пользователей

**Название:** `user_status`

| Колонка | Тип | Описание |
|---|---|---|
| `chat_id` | text | ID чата пользователя в Telegram |
| `last_command` | text | Последняя команда (`/form_problem` или `request_submitted`) |
| `timestamp` | number | Unix timestamp последнего обновления |

**Используется в узлах:**
- `Check User State` — чтение состояния по chat_id
- `Record Command Usage` — запись при использовании `/form_problem`
- `Update User State After Request` — обновление после обработки задачи

---

#### Таблица 2: Логирование ответов

**Название:** `analytics` (или любое другое — задаётся в узле `Log to Data Table`)

| Колонка | Тип | Описание |
|---|---|---|
| `chat_id` | text | ID чата пользователя |
| `user_id` | text | ID пользователя в Telegram |
| `timestamp` | text | Временная метка |
| `formatted_response` | text | Полный сгенерированный план (Markdown) |
| `review_data` | text | Данные Review Plan (score + comment) |
| `offline_score` | text | Суммарный балл Offline Evaluator |
| `offline_verdict` | text | Вердикт Offline Evaluator |
| `offline_match_quality` | text | Качество совпадения с датасетом |
| `offline_comment` | text | Комментарий Offline Evaluator |
| `offline_scores` | text | Детальные оценки по критериям (JSON) |

**Используется в узле:** `Log to Data Table`

---

### Привязка таблиц к узлам

После создания таблиц откройте каждый узел и привяжите таблицу:

| Узел | Таблица |
|---|---|
| `Check User State` | `user_status` |
| `Record Command Usage` | `user_status` |
| `Update User State After Request` | `user_status` |
| `Log to Data Table` | `analytics` |

---

### Шаг 5: Активация workflow

1. Убедитесь, что все Telegram credentials настроены
2. Убедитесь, что все Mistral AI credentials настроены во всех 9 узлах
3. Убедитесь, что обе Data Tables созданы и привязаны к узлам
4. Нажмите переключатель **Active** в правом верхнем углу

### Сценарий 1: Первое сообщение без команды

**Действие:** Отправьте боту любое текстовое сообщение, например `Привет`

**Ожидаемый результат:**
> ⚠️ Пожалуйста, сначала используйте команду /form\_problem

---

### Сценарий 2: Команда /start или /help

**Действие:** Отправьте `/start` или `/help`

**Ожидаемый результат:** Бот отправит описание своих возможностей и примеры запросов

---

### Сценарий 3: Команда /form_problem

**Действие:** Отправьте `/form_problem`

**Ожидаемый результат:**
- Бот отправит описание возможностей
- Запросит описание аналитической задачи

---

### Сценарий 4: Корректный запрос (основной сценарий)

**Действие:** После `/form_problem` отправьте:
```
Нужно понять, почему упала конверсия в покупку на 15% за последний месяц в мобильном приложении
```

**Ожидаемый результат:**
1. Бот отвечает `⏳ Начинаю анализ...`
2. Через 30–90 секунд отправляет полный план анализа из 7 разделов
3. Если ответ длинный — разбивает на несколько сообщений

---

### Сценарий 5: Нечёткий запрос

**Действие:** Отправьте короткое неинформативное сообщение: `анализ` или `помоги`

> ❓ ТРЕБУЕТСЯ УТОЧНЕНИЕ  
> Ваш запрос недостаточно конкретен...  
> [подсказка от LLM с конкретными вопросами]

## Мониторинг и логирование

### Просмотр выполнений

1. Откройте workflow в n8n
2. Перейдите во вкладку **Executions**
3. Выберите любое выполнение для детального просмотра

Каждое выполнение показывает входные/выходные данные всех 52 узлов, время выполнения и статус.

### Просмотр логов ответов

1. Перейдите в **Data Tables** → таблица `analytics`
2. Каждая строка — один обработанный запрос с полным планом, оценками обоих эвалюаторов и метаданными пользователя

### Просмотр состояний пользователей

1. Перейдите в **Data Tables** → таблица `user_status`
2. Каждая строка — текущее состояние пользователя (last_command + timestamp)

### Ключевые метрики для мониторинга

- **Review Score** (`review_data.score`) — оценка качества плана от 0 до 10
- **Offline Score** (`offline_score`) — оценка Offline Evaluator от 0 до 10
- **Offline Verdict** (`offline_verdict`) — `excellent` / `good` / `acceptable` / `weak` / `bad`
- Время выполнения полного пайплайна (в Executions)

## Возможные проблемы и решения

### Бот не отвечает на сообщения

- Проверьте, что workflow **активирован** (переключатель Active включён)
- Проверьте **Telegram credentials** в узле Telegram Trigger
- Убедитесь, что webhook Telegram настроен корректно (n8n должен быть доступен по публичному URL)

---

### Ошибка `Cannot read properties of undefined`

- Проверьте, что обе **Data Tables** созданы с правильными именами колонок
- Проверьте привязку таблиц к узлам (`Check User State`, `Record Command Usage`, `Update User State After Request`, `Log to Data Table`)
- Убедитесь, что колонка `last_command` существует в таблице `user_status`

---

### Ошибка Mistral AI / пустой ответ агента

- Проверьте **API ключ** в credentials Mistral AI
- Проверьте лимиты и баланс на [console.mistral.ai](https://console.mistral.ai)
- Убедитесь, что модель `mistral-large-latest` доступна в вашем аккаунте
- Убедитесь, что credentials привязаны ко **всем 9 узлам** Mistral Model

---

### Ошибка JSON парсинга (Parse Review Result / Offline Evaluator Schema)

- Проверьте, что в системных промптах агентов `Review Plan` и `Offline Evaluator` явно прописано требование возвращать только JSON
- Если ошибка возникает редко — это нормально; Mistral иногда добавляет лишний текст. Узел `Parse Review Result` содержит try/catch для таких случаев

---

### Бот не запоминает состояние между сессиями

- Проверьте таблицу `user_status` — есть ли запись для нужного `chat_id`
- Проверьте, что узлы `Record Command Usage` и `Update User State After Request` успешно выполняются (смотрите Executions)
- Убедитесь, что `chat_id` корректно извлекается в узле `Extract Message Data`

---

### Ответ не разбивается на части при длинном тексте

- Проверьте узел **Split Long Message**: он делит текст по строкам при превышении 3800 символов
- Убедитесь, что узел **Send to Telegram** итерирует по всем чанкам из Split Long Message

## Технические детали

| Параметр | Значение |
|---|---|
| Платформа | n8n (self-hosted или cloud) |
| AI провайдер | Mistral AI |
| Модель (Frame Problem) | `ministral-8b-latest` |
| Модель (остальные агенты) | `mistral-large-latest` (рекомендуется) |
| Количество узлов | 52 |
| Количество LLM-агентов | 9 (8 пайплайн + 1 эвалюатор) |
| Максимальная длина сообщения | 3800 символов (автосплит) |
| Язык ответа | Автоматически — язык запроса пользователя |
| Язык системных промптов | Английский (агенты пайплайна), Русский (Review, Offline Evaluator) |
| Формат ответа | Markdown |
| Хранение состояния | n8n Data Tables |
| Логирование | n8n Data Tables (`analytics`) |
